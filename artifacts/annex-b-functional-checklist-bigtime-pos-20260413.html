BIGTIME POS

Accomplished Standard Functional and Technical Requirements Form (Annex "B")

Prepared on: April 13, 2026
System name: BIGTIME POS
System coverage: Dashboard, central API, and Android/Windows POS terminal application
Document status: Internal accomplished working draft for Annex "B" review

Purpose

This document is a working draft of the Standard Functional and Technical Requirements form (Annex "B") for BIGTIME POS. It is based on:

  • the official BIR Annex "B" checklist
  • the BIR accreditation framework under RMO No. 24-2023
  • the current BIGTIME POS codebase, system documentation, and observable application behavior as of April 13, 2026

This draft is intended to help prepare the final sworn statement package, system demonstration, and supporting evidence bundle. Final sign-off should still be based on:

  • deployed production configuration
  • actual printed receipts and reports
  • final BIR settings per branch/terminal
  • documentary and operational controls outside the software itself

Legend

  • YES - feature is implemented and demonstrable in the current system
  • PARTIAL - feature is present in part, is configuration-dependent, or still needs final template/policy validation
  • NO - feature is not yet verified or not implemented in the current build
  • N/A - not applicable to the present scope of BIGTIME POS as a POS and back-office platform

1. System-Generated Receipts / Invoices

Description Complied Remarks
a. Seller's Registered Name YES Receipt header supports registered/store name pulled from branch and BIR settings.
b. Seller's Business Name / Business Style YES Store name and proprietor/business identifiers are supported in the receipt header configuration.
c. Seller's Registered Address where machine will be used YES Business address lines are supported in receipt header settings.
d. VAT REG TIN or NON-VAT REG TIN followed by TIN and branch code PARTIAL The receipt template supports the TIN line. Final nine-digit TIN plus five-digit branch code formatting depends on branch BIR configuration and print validation.
e. Serial Number with minimum six running digits and numbering convention PARTIAL Official OR/serial issuance is centrally controlled by the server. Final zero-padded six-digit numbering presentation should be finalized and validated in the print template.
f. Date of Transaction YES Receipt output includes transaction date and time.
g. Space for Customer/Buyer details: registered name, address, TIN, branch code, business style PARTIAL Standard retail flow does not yet expose all buyer fields on every printed output. Invoice-mode buyer capture and layout validation are still needed where required.
h. Quantity YES Quantity is printed for each sold line item.
i. Description of items sold / nature of services YES Item description is printed in the receipt body.
j. Unit Cost / SRP YES Unit price is printed per line item.
k. Total Cost / Amount YES Line totals and total amount are generated and printed.
l. VAT Amount YES VAT amount is reflected for taxable transactions.
m. Mixed transaction breakdown: VATable, VAT, VAT Exempt, Zero-Rated PARTIAL VATable sales and VAT totals are already reflected. VAT-exempt and zero-rated breakdown lines still need final mixed-transaction print validation.
n. Word EXEMPT shown for exempt-only transactions PARTIAL Exempt tax handling exists in pricing configuration, but final receipt wording and exempt-only format should be validated in the branch receipt template.
o. SC/PWD details: TIN, ID no., discount breakdown, signature space PARTIAL Senior Citizen and PWD discounts exist in pricing rules, but the full receipt layout for ID/TIN/signature and detailed statutory breakdown still needs final validation.
p. Bottom print details: acknowledgment certificate, date issued, series range, five-year validity phrase, input tax phrase PARTIAL AC/PN, MIN, SN, and the five-year validity phrase are supported. Explicit date issued, series range, and the supplementary/input-tax wording should be finalized to the exact BIR format.
q. Word REPRINT on duplicate or subsequent printouts YES Reprint flow applies copy labeling and requires a reprint reason.

2. Duplicate Invoice / Receipt Preservation

Description Complied Remarks
Commercial invoices are prepared at least in duplicate; original is issued and duplicate is preserved PARTIAL Original issuance is supported and duplicate electronic retention exists through server and local transaction records. Physical preservation of duplicates remains an operational/store procedure.

3. System-Generated Books of Accounts

Description Complied Remarks
a. General Journal mandatory fields under RR No. 9-2009 PARTIAL Ledger and reporting foundations exist, but the final General Journal output format should still be validated against the RR No. 9-2009 field layout.
b. General Ledger mandatory fields under RR No. 9-2009 PARTIAL The system stores transaction-ledger data, but final General Ledger presentation and export layout still require Annex "B" format validation.
c. Sales Journal mandatory fields under RR No. 9-2009 PARTIAL Sales, OR/reference, discounts, VAT, and totals are available. Buyer-specific fields depend on invoice-mode data capture and final report template validation.
d. Purchase Journal mandatory fields under RR No. 9-2009 PARTIAL Supplier, purchase-order, and inventory modules exist in the back office, but final Purchase Journal output validation is still needed.
e. Inventory Book mandatory fields under RR No. 9-2009 PARTIAL Inventory and stock modules are available, but the final Inventory Book report/export format should still be standardized for Annex "B".

4. Printout and Electronic Copy of Reports

Description Complied Remarks
a. Taxpayer's Registered Name on reports PARTIAL Report modules exist in the dashboard, but standardized report header output still needs final print/export validation.
b. Registered Address where such reports are generated PARTIAL Address support exists through branch and BIR settings, but report header standardization still needs confirmation.
c. VAT REG TIN or NON-VAT REG TIN with seller TIN and branch code PARTIAL Data source exists through BIR settings. Report print/export templates should still be finalized to the exact required format.
d. Software Name and Version Number PARTIAL Software identity and version can be supplied, but final placement on report printouts and exports still needs standardization.
e. User Name / ID of the one who generated the report PARTIAL Authenticated report access exists, but printed/exported report attribution needs final template validation.
f. Date and Time Stamp when report was generated PARTIAL Generated timestamps exist in the system, but final display on printed/exported report forms should still be validated.

5. Saving Reports in .csv or .dat

Description Complied Remarks
The system is capable of saving Books of Accounts, Financial Statements, and other reports in .csv or .dat format PARTIAL Dashboard reporting and data modules are present, but the final RR No. 16-2006 compliant export package for all required report types still needs final validation.

6. Sales Data Transmission to the BIR

Description Complied Remarks
The system can send sales data / information to the BIR in the mandated format YES BIGTIME POS includes EIS submission, signing, retry, acknowledgment tracking, and BIR EIS monitoring endpoints. Live filing is available once production endpoint and credentials are configured.

7. Database Backup Retention

Description Complied Remarks
The system database backup can be preserved for the mandatory ten-year retention period PARTIAL Central and local data stores are persistent and backup procedures are documented. Actual ten-year retention depends on implemented production backup, archive, and storage policy.

8. Audit Trail / Activity Log

Description Complied Remarks
The system generates an audit trail or activity log of transactions and activities that can be printed PARTIAL BIGTIME POS maintains audit and receipt-event logs, including hash-linked critical records. Query and review are available; final print/export presentation should still be standardized.

9. BIR Withholding Tax Certificates

Description Complied Remarks
Generation of BIR Form 2306, 2307, and 2316 N/A BIGTIME POS is currently positioned as a POS and back-office sales platform. Withholding tax certificate generation is outside the present sales issuance scope of this build.

10. Complete Audit Trail Functionalities

Description Complied Remarks
a. System controls the receipt numbering YES Official OR numbering is server-issued and centrally controlled by the POS ledger service.
b. Posting date of receipt transaction is the date it entered the system YES Transactions are stored with persistent created/posting timestamps at the time of entry.
c. Transaction can be voided, but not modified once posted YES Posted sales are preserved in append-only fashion. Adjustments use controlled void/refund workflows instead of direct record editing.
d. Run date is the date the report was generated YES Sales summaries, end-of-day reports, and related outputs carry generation timestamps.
e. Users are prevented from editing data within system-generated reports PARTIAL End users do not directly edit generated outputs in the current flow, but readonly export controls should still be validated in the final dashboard report package.
f. Users are prevented from having the capability to override edits within computer programs YES Application actions are role-protected and critical transaction records are stored through guarded service flows rather than ad hoc direct editing.
g. Each transaction / record is stamped with the user ID of the individual that created the transaction YES Cashier IDs and user IDs are stored in transactions, sessions, and audit entries.
h. System totals and cross-checks totals and reports incorrect computation or out-of-balance condition YES The system computes subtotal, VAT, discounts, tendered amount, change, sales summaries, and Z-readings programmatically.

11. System Security Framework

Description Complied Remarks
a. Standard procedure and approval in securing user access PARTIAL Authenticated access, roles, and account provisioning exist. Formal written provisioning/approval SOP should still be attached as an operational control.
b. Users are restricted from being active on multiple terminals at the same time with the same identification code YES Active session tracking revokes prior active sessions for the same user upon new login.
c. User identification codes are deactivated after unsuccessful attempts to sign on NO Failed-login lockout / auto-deactivation control is not yet verified in the current build.
d. Password is required to be changed every thirty (30) days NO Forced thirty-day password or PIN rotation is not yet enforced in the current build.
e. Non-admin users are prevented from accessing sensitive operating system commands PARTIAL Application roles block sensitive business actions, but OS-level workstation lockdown depends on device hardening and operational setup.
f. Passwords require a combination of alphabetic and numeric characters PARTIAL Web/admin access can support stronger credentials, but the cashier terminal presently uses PIN-based authentication instead of an alphanumeric-only policy.
g. Users are prevented from accessing accounting information except through authorized transactions YES Protected resources are guarded by JWT authentication, throttling, and role-based authorization.
h. User access rights can be eliminated or revised upon termination or transfer PARTIAL Employee/user administration and inactive-account handling exist, but HR offboarding and reassignment remain operational processes.
i. Computerized audit trails are protected from modification and destruction YES Critical audit records are hash-chained, and destructive reset operations are disabled in compliance mode.
j. Server rooms and data centers are locked and access-controlled PARTIAL This requirement depends on the selected hosting facility or on-premise deployment controls rather than code alone.
k. Server rooms and data centers are equipped with air-conditioning PARTIAL Environmental control is a deployment-site requirement outside the application code.
l. Server rooms and data centers are protected against power failure and fire PARTIAL This is dependent on UPS, fire-suppression, and facility controls in the production environment.
m. Server room and data center cables are properly installed PARTIAL Cabling standard is a physical infrastructure requirement to be satisfied by the deployment environment.
n. Unused network connection points are disabled PARTIAL Network hardening is handled at the infrastructure layer and should be addressed during deployment.
o. All incoming dial-up uses strong password authentication N/A Dial-up connectivity is not part of the BIGTIME POS architecture.
p. Remote access via Internet is encrypted PARTIAL Production remote access can be protected with HTTPS/TLS and secure network configuration, but this must be enforced in deployment.
q. Access to external networks occurs over a firewall and is monitored and audited PARTIAL Firewalling and monitoring depend on the final hosting/network stack and should be documented operationally.
r. Application access is protected with an authentication mechanism YES Application login, JWT/session handling, and role-based access are implemented.
s. Database record modification is logged for critical applications PARTIAL Critical sales, receipt, shift, and audit events are logged. Full database-wide change logging for every non-critical master table is not yet being claimed in this draft.

12. Adjustment to Sales / Revenue

Description Complied Remarks
a. There is a mechanism that enables adjustment to sales / revenue, such as supplementary documents generated and issued to other parties PARTIAL BIGTIME POS supports void and refund workflows with reason capture and audit logging. Final supplementary document templates and business process notes should still be standardized.
b. Supplementary documents must have a unique serial number PARTIAL Void/refund references and transaction IDs are unique, but a formal supplementary document numbering convention should still be documented for submission.

Overall Assessment

BIGTIME POS already demonstrates a strong compliance foundation for the Annex "B" checklist in the following areas:

  • controlled receipt issuance and server-side OR handling
  • sales, VAT, discount, and payment computation
  • audit trail and receipt-event logging
  • void/refund adjustment workflows
  • Z-reading and BIR/EIS monitoring capabilities
  • authenticated access with session tracking and role-based protection
  • central API plus local POS storage for operational continuity

The main items still requiring final validation, policy attachment, or template refinement are:

  • buyer/invoice fields on standard receipt outputs where applicable
  • exact BIR receipt footer formatting such as series range and date issued
  • final Books of Accounts and report export templates
  • long-term backup retention policy and hosting controls
  • specific security policies such as failed-login lockout and credential rotation
  • physical infrastructure controls outside the application code
