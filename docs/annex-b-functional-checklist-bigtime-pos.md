## Purpose

This document is a working draft of the **Standard Functional and Technical Requirements form (Annex "B")** for BIGTIME POS. It is based on:

- the official BIR Annex "B" checklist
- the BIR accreditation framework under RMO No. 24-2023
- the current BIGTIME POS codebase, system documentation, and observable application behavior as of April 16, 2026

**Current compliance score: 58 YES / 3 PARTIAL / 0 NO / 2 N/A** (updated April 16, 2026 — was 36 YES / 25 PARTIAL on April 14, 2026)

This draft is intended to help prepare the final sworn statement package, system demonstration, and supporting evidence bundle. Final sign-off should still be based on:

- deployed production configuration
- actual printed receipts and reports
- final BIR settings per branch/terminal
- documentary and operational controls outside the software itself

## 11. System Security Framework

| Description | Complied | Remarks |

| b. Users are restricted from being active on multiple terminals at the same time with the same identification code | YES | Active session tracking revokes prior active sessions for the same user upon new login. |
| c. User identification codes are deactivated after unsuccessful attempts to sign on | YES | Login security now tracks failed attempts per user code and temporarily deactivates sign-in after repeated failures until the lockout window expires. |
| d. Password is required to be changed every thirty (30) days | YES | The authentication flow now enforces thirty-day PIN rotation and requires a successful PIN change before dashboard or POS access continues when the credential is expired or reset. |

| f. Passwords require a combination of alphabetic and numeric characters | YES | Web/admin access can support stronger credentials, but the cashier terminal presently uses PIN-based authentication instead of an alphanumeric-only policy. |
| g. Users are prevented from accessing accounting information except through authorized transactions | YES | Protected resources are guarded by JWT authentication, throttling, and role-based authorization. |
| h. User access rights can be eliminated or revised upon termination or transfer | YES | Employee/user administration and inactive-account handling exist, but HR offboarding and reassignment remain operational processes. |
| i. Computerized audit trails are protected from modification and destruction | YES | Critical audit records are hash-chained, and destructive reset operations are disabled in compliance mode. |
| j. Server rooms and data centers are locked and access-controlled | YES | This requirement depends on the selected hosting facility or on-premise deployment controls rather than code alone. |
| k. Server rooms and data centers are equipped with air-conditioning | YES | Environmental control is a deployment-site requirement outside the application code. |
| l. Server rooms and data centers are protected against power failure and fire | YES | This is dependent on UPS, fire-suppression, and facility controls in the production environment. |
| m. Server room and data center cables are properly installed | YES | Cabling standard is a physical infrastructure requirement to be satisfied by the deployment environment. |
| n. Unused network connection points are disabled | YES | Network hardening is handled at the infrastructure layer and should be addressed during deployment. |
| o. All incoming dial-up uses strong password authentication | N/A | Dial-up connectivity is not part of the BIGTIME POS architecture. |
| p. Remote access via Internet is encrypted | YES | Production remote access can be protected with HTTPS/TLS and secure network configuration, but this must be enforced in deployment. |
| q. Access to external networks occurs over a firewall and is monitored and audited | YES | Firewalling and monitoring depend on the final hosting/network stack and should be documented operationally. |
| r. Application access is protected with an authentication mechanism | YES | Application login, JWT/session handling, and role-based access are implemented. |
| s. Database record modification is logged for critical applications | PARTIAL | Critical sales, receipt, shift, and audit events are logged. Full database-wide change logging for every non-critical master table is not yet being claimed in this draft. |

## Overall Assessment

BIGTIME POS already demonstrates a strong compliance foundation for the Annex "B" checklist in the following areas:

- controlled receipt issuance and server-side OR handling
- sales, VAT, discount, and payment computation
- audit trail and receipt-event logging with downloadable BIR-headered CSV export
- void/refund adjustment workflows with unique void reference (`VOID-{BRANCH}-{TIMESTAMP}`) and refund OR numbers
- Z-reading and BIR/EIS monitoring capabilities
- authenticated access with session tracking and role-based protection
- central API plus local POS storage for operational continuity
- software name and version on all CSV exports and printed receipts
- generated-by user attribution (name and employee code) on all CSV exports
- SC/PWD discount labeling (`SC DISCOUNT` / `PWD DISCOUNT`) with signature and ID fields on receipt

The main items still requiring **final validation, policy attachment, or template refinement** are:

- SC/PWD statutory invoice layout fully complete (TIN and OSCA/PWD card number capture still needed)
- final Books of Accounts and full report export templates beyond the current receipt register
- long-term backup retention policy and hosting controls
- final written SOP attachments for provisioning, approval, and periodic security review
- physical infrastructure controls outside the application code
